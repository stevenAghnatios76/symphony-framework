01

Installation & setup

Symphony runs inside your AI coding tool of choice. Install it once, configure a few fields in global.yaml, and every workflow is immediately available as a slash command.

1
Clone or install the framework
Add Symphony to your project root or to a shared directory.
# Option A — add to an existing project git clone https://github.com/your-org/symphony-framework .symphony-fw cp -r .symphony-fw/_symphony ./ rm -rf .symphony-fw # Option B — install via npm npm install symphony-framework npx symphony-framework --version
2
Pick your adapter
Symphony has a thin adapter for each AI tool. Each adapter generates the tool-specific command files from the same Core — no business logic lives in the adapter.
# Claude Code (default) # Adapter writes slash commands to .claude/commands/symphony-*.md # Set up via CLAUDE.md — Symphony is auto-discoverable # GitHub Copilot # Adapter writes .github/prompts/symphony-*.prompt.md # + copilot-instructions.md for global context # Gemini CLI · Cursor — registered, same pattern
3
Edit _symphony/_config/global.yaml
This is the only file you need to change before your first run. Set your name, project name, and where your source lives.
framework_name: "Symphony" framework_version: "0.0.2-alpha.2" user_name: "Alice" # your name project_name: "my-app" # your project name project_path: "." # "." = root, or "src" if code lives in a subdir
Tip That's it. You don't need to touch limits, disciplines, or vault settings to get started. The defaults are production-ready.
4
Choose an environment preset (optional)
Three presets scale the gate strictness to match your team size. Copy the one that fits into global.yaml.
# _symphony/_config/presets/ solo.yaml # no gates, single-agent, weekly memory hygiene team.yaml # code review gate, multi-agent, daily hygiene enterprise.yaml # all 6 gates + compliance audit trail
02

The core concept

Symphony is not a chatbot wrapper. It's an orchestration engine. You describe a goal; the Conductor figures out where you are in the lifecycle and dispatches the right agents.

→
You don't drive — you declare
Instead of copying prompts or managing agents manually, you run a single slash command. Symphony handles sequencing, parallelism, memory, and gate enforcement. Your job is to approve outputs at gate checkpoints.
→
Phases produce artifacts, not just answers
Each workflow writes structured outputs to your project: docs/brief.md, docs/prd.md, docs/architecture.md, sprint files, story files. These artifacts form a complete traceability chain — every line of code traces back to a requirement.
→
The engine enforces quality — you don't have to
Self-critique at ≥ 0.85 confidence, anti-rationalization checks, trust levels, and HALT gates are baked into the engine. Agents cannot skip them. If a gate fails, work stops and the problem is surfaced — never silently swallowed.
Key idea Confidence ≥ 0.80 = Conductor auto-dispatches. < 0.80 = it asks you to pick. You only need to intervene when Symphony is genuinely unsure about context.

Choosing your entry point

The Conductor auto-detects which path to take based on what artifacts already exist. You can also invoke any path explicitly.

When to use
Greenfield — New Project
Starting from scratch. No code, no brief, no PRD. You have an idea and want Symphony to take you all the way through.
Starts at → Phase 1 (Analysis)
When to use
Overture — Existing Codebase
You have code but no Symphony artifacts. Overture scans in parallel waves, maps your architecture, and generates a gap-focused PRD.
Starts at → Phase 3 (Solutioning)
When to use
Add Feature — Cross-Phase Cascade
Project is running, you need to add a feature. Symphony cascades the change through PRD → Architecture → Test Plan → Stories → Trace.
Starts at → P2 → P3 → P4
When to use
Quick Path — Small Change
A bug fix, a small UI tweak, a config change. Full lifecycle is overkill. Quick-spec + quick-dev handles it in two workflows.
Starts at → quick-spec → quick-dev
When to use
Debug / Fix
Something is broken. The diagnose-then-fix protocol forces root-cause analysis before any code changes. Up to 3 retry attempts before escalation.
Starts at → diagnose-then-fix protocol
When to use
Just tell Symphony what you want
Type a natural language goal in your AI tool. The Conductor reads it, scores confidence, and routes automatically. No command memorization required.
Routed by → Conductor (≥ 0.80 confidence)
Auto-detection rules No artifacts at all → Phase 1. Source code exists but no Symphony docs → Overture (auto). Brief exists but no PRD → Phase 2. PRD exists but no arch → Phase 3. Architecture + active sprint → Phase 4. Everything done, not shipped → Phase 5.
04

The 5 lifecycle phases

Each phase produces a set of artifacts that gate the next phase. You move forward when a gate passes — never before.

Phase 01
Analysis
Explore the problem space. Produce a product brief — the origin of the traceability chain. Gate: brief approved.
brainstorm market-research domain-research tech-research advanced-elicitation product-brief ← gate
Phase 02
Planning
Translate the brief into a PRD, UX spec, and an accessibility review. PRD validation loop ensures consistency before moving to architecture.
create-prd validate-prd ← gate edit-prd create-ux review-a11y
Phase 03
Solutioning
Design the technical solution. Architecture, test strategy, epics, ATDD, security threat model, infra design, and a full traceability matrix. Gate: all artifacts ready.
create-arch test-design create-epics atdd threat-model infra-design trace readiness-check ← gate
Phase 04
Implementation
Sprint-based delivery loop. Create → validate → plan → develop → DoD check → 6-gate review → gate. Repeat per story until sprint is complete, then retro.
create-story validate-story sprint-plan dev-story check-dod run-all-reviews ← 6 gates review-gate ← HALT sprint-status retro
Phase 05
Deployment
Release planning, rollback strategy, pre-deploy checklist, and post-deploy verification. Terminal state — the lifecycle is complete.
release-plan rollback-plan deploy-checklist ← gate post-deploy ← terminal
05

How the Conductor works

The Conductor is Symphony's routing brain. It reads your goal, scans artifact state, scores a confidence level, and dispatches — or asks you to choose if it's not sure.

1
Goal parsing
The Conductor extracts intent from natural language. Keywords like fix, debug, error → diagnose-then-fix. Keywords like onboard, scan, analyze existing → Overture. No keyword match → artifact-state detection.
2
Artifact-state detection
The Conductor looks for landmark files to determine lifecycle position:
# No brief.md → Phase 1 # brief.md exists, no prd.md → Phase 2 # prd.md exists, no architecture.md → Phase 3 # architecture.md exists, active sprint → Phase 4 # All complete, not shipped → Phase 5 # Source code exists, no _symphony/state/ → Overture
3
Confidence scoring & dispatch
The Conductor assigns a confidence score (0.0–1.0) to its routing decision.
Routing rule Score ≥ 0.80 → auto-dispatch the workflow.
Score < 0.80 → present options, ask you to confirm.
The routing memory sidecar logs every routing decision. When you correct a wrong routing, the Conductor updates its memory and improves future decisions on similar goals.
06

Execution & interaction modes

Symphony has two independent mode axes. You can set any combination of execution mode + interaction mode.

→
Execution modes — how agents run
Sequential
Agents run one at a time, each waiting for the previous. Default for most lifecycle workflows.
Parallel Waves
DAG-sorted. Up to 4 agents run concurrently per wave. Integration gates between waves catch conflicts. Used by Overture and run-all-reviews.
Ensemble
Multiple agents share a conversation and take turns. Turn policies: round-robin, facilitator-picks, user-picks. Used by brainstorm, retro, creative workflows.
→
Interaction modes — when you're involved
Normal
Pauses at each checkpoint for your review and approval. Recommended for first-time use and critical phases.
YOLO
Auto-proceeds through checkpoints without pausing. Gates still HALT — YOLO never skips a quality gate, only confirmation prompts.
Planning
Shows the full execution plan — all steps, agents, and gates — before any work begins. You approve the plan, then execution starts.
Important YOLO mode does not bypass gates. If a gate fails in YOLO mode, execution halts and requires your intervention — same as Normal mode.

Agents reference

You never invoke agents directly — the Conductor and workflow engine dispatch them. But knowing who does what helps you understand what you're reviewing at each gate.

product-manager
Owns briefs, PRDs, epics, and stories. Defines requirements and acceptance criteria throughout the lifecycle.
architect
Designs system architecture — components, APIs, data models, integration patterns.
ux-designer
Produces UX specs, wireframes, user flows, and interaction patterns from the PRD.
developer
Implements stories end-to-end using TDD. Detects your stack (TypeScript, Flutter, Python, Go, Angular) and loads matching knowledge JIT.
test-architect
Designs test strategy, runs ATDD, and manages the 30-fragment testing knowledge base across 6 categories.
security-agent
Runs STRIDE threat modeling in Phase 3 and OWASP/dependency scanning during the 6-gate review.
devops-agent
Handles infra design, CI/CD, release plans, rollback strategy, and post-deploy verification.
reviewer
Code quality review — patterns, best practices, maintainability. One of the 6 gates in run-all-reviews.
research-analyst
Market research, domain research, technology evaluation. Active in Phase 1 (Analysis).
critic
Adversarial agent — challenges assumptions, devil's advocate reviews, surfaces blind spots.
code-simplifier
Refactoring specialist. Reduces complexity, enforces YAGNI, improves readability without changing behavior.
browser-tester
Playwright E2E tests, visual regression, accessibility audits, responsive testing across viewports.
mobile-dev
React Native, Flutter, Kotlin, Swift — 8-step TDD protocol applied to mobile stacks.
designer-mobile
iOS Human Interface Guidelines + Material Design 3. Produces platform-native design specs.
mobile-tester
Device matrix coverage, Detox/Maestro/Appium test plans, platform-specific verification.

Tips & common mistakes

Things that trip up new Symphony users — and what to do instead.

Don't skip gates
Gates exist to catch problems early. If a gate fails, fix the underlying issue — don't look for ways around it. Skipped gates always surface as bigger problems later.
Use memory hygiene weekly
Run /symphony-memory-hygiene at the end of each sprint. Stale decisions in agent sidecars cause incorrect behavior in future workflows and are hard to debug.
Start with Normal mode
Use Normal interaction mode until you understand the rhythm of gate checkpoints. Switch to YOLO only on phases you've run before and understand well.
Correct the Conductor explicitly
If the Conductor routes to the wrong phase, tell it explicitly: "No, I want to start at Phase 2." It logs the correction and improves future routing on similar goals.
Don't edit _symphony/ files manually
Agent, workflow, and protocol files are read by the engine at activation time. Manual edits may break internal references or violate the engine's validation rules.
Overture for any existing codebase
Even if you wrote the code yourself, run Overture on an existing project before starting Symphony workflows. It generates the artifact baseline that all subsequent phases depend on.
The Quick Path is genuinely quick
For fixes, small features, and config changes: /symphony-quick-spec then /symphony-quick-dev. Two commands, no PRD, no architecture — done in minutes.
YOLO ≠ skip gates
This is the most common misunderstanding. YOLO auto-approves checkpoints but cannot skip a quality gate. If code fails review, YOLO mode still halts — exactly like Normal mode.
