Symphony's Unique Advantages — What No Other Framework Has
Smart Conductor with Confidence Gating
Parses natural language goals, detects project phase from artifact state, auto-routes with confidence scoring (≥0.80). Learns from corrections via routing memory sidecar.
vs Gaia: manual /gaia-command selection | vs GSD: manual /gsd-command | vs Gem: basic phase detection, no confidence scoring
DAG-Based Parallel Wave Execution
Topologically sorts step dependencies, dispatches ≤4 parallel subagents per wave, runs integration gates at wave boundaries with conflict detection.
vs All: No other framework has true DAG-based parallelism with integration gates
Ensemble Mode (Multi-Agent Discussion)
First-class execution mode where multiple agents take turns in a shared conversation with the user as active participant. Turn policies: round-robin, facilitator-picks, user-picks.
vs Gaia: /gaia-party is limited | vs GSD: /gsd-discuss is single-agent | vs Gem: no persistent discussion
3 Interaction Modes (Normal / YOLO / Planning)
Orthogonal to execution mode. YOLO auto-proceeds but gates still HALT. Planning shows full execution plan before any work begins.
vs All: Other frameworks only support normal mode
Anti-Rationalization Protocol
Every agent carries an Excuse → Rebuttal table. Engine-enforced, not optional. Catches "this edge case is rare" and similar rationalizations.
vs All: No other framework has systematic anti-rationalization
Engine-Enforced Self-Critique
Every agent output passes through confidence threshold (≥0.85). Below threshold: retry once, then escalate. Not opt-in — the engine enforces it.
vs Gaia: no self-critique | vs GSD: no self-critique | vs Gem: has confidence 0.85 threshold per agent + gem-critic — but Symphony's is engine-enforced, not per-agent opt-in
Adaptive Quality Gates per Story Type
6-gate review process auto-adapts based on story's traces_to prefix. Infrastructure stories get IaC-specific gates (checkov, OPA, cost review) instead of standard tests.
vs All: Other frameworks use one-size-fits-all review gates
Dual-Layer Adapter Architecture
Core is tool-agnostic. Adapters are pure file generators with zero business logic. New AI tools = afternoon of adapter work, not a framework refactor.
vs Gaia: Claude-only | vs GSD: multi-tool but logic duplicated per tool | vs Gem: Gemini-only

Improvement Roadmap — What to Absorb from Each Framework
Absorb from Gaia
  • HIGH Full creative module — design thinking, innovation strategy, storytelling, slide/pitch decks, creative sprints. Symphony's ensemble mode makes these even more powerful.
  • HIGH Deep testing suite — ATDD, test gap analysis, a11y testing, performance testing, mobile testing, test framework setup, CI/CD setup. These fill Symphony's testing module.
  • MED Lifecycle-sequence branching logic — on_pass/on_fail routing, primary/alternatives, standalone workflows. Implement as a sequence manifest in Symphony.
  • MED Party mode (stakeholder discussions) — absorb as an ensemble-mode workflow with stakeholder personas.
  • LOW Quick flow (/gaia-quick-spec + /gaia-quick-dev) — already planned as /symphony-quick-spec + /symphony-quick-dev.
  • LOW Agent count depth (31+ vs ~15) — add specialist agents: editorial reviewer, changelog generator, adversarial agent.
Absorb from GSD
  • MED Multi-tool breadth — GSD supports 10+ tools. Symphony's adapter model is cleaner but needs more adapters: Gemini CLI, Cursor, Codex, Kilo, Trae, Cline.
  • MED Workspace isolation — /gsd-new-workspace creates isolated copies with independent state. Useful for feature isolation and multi-repo work.
  • MED Fresh context per agent — GSD's 200K fresh context window per subagent eliminates context rot. Consider as a wave executor enhancement.
  • LOW CLI tools layer (gsd-tools.cjs) — unified tool for state management, config, phase tracking, roadmap, verification, templates. Symphony could benefit from similar CLI tooling.
  • LOW Internationalization — GSD has docs in Japanese, Korean, Chinese, Portuguese. Consider for Symphony's global reach.
Absorb from Gem Team
  • MED Gray area detection (discuss phase) — Gem's orchestrator detects APIs, visual features, business logic, and data format ambiguities, then generates context-aware multiple-choice questions. Add to Symphony's Conductor.
  • MED Concurrent research delegation — Gem spawns up to 4 concurrent researchers per focus area. Symphony's wave executor already supports this; wire it up for research workflows.
  • LOW PRD.yaml structured format — Gem uses YAML for PRDs with explicit IN SCOPE, OUT OF SCOPE, NEEDS CLARIFICATION sections. Consider as template option.
  • LOW Complexity classification (simple/medium/complex) — Gem auto-classifies task complexity to decide research depth. Could enhance Symphony's Conductor routing.
  • LOW Architectural decisions auto-append to AGENTS.md — useful convention for persistent decision capture beyond sidecars.
